Workflow Process Summary



A task is a single, self-contained prediction challenge built on top of a dataset you know well. 



For every task:

The agent will be asked to train a model / pipeline / etc on train.csv

The agent uses the model to predict test.csv

The agent generates the predictions as submission.csv

We run score_submission.py to validate and score the submission deterministically against solution.csv

We will run `python3 score_submission.py --submission-path submission.csv --solution-path solution.csv`

We expect this command to output a single float that represents the score

Reviewers check the task is correct + realistic + scorable

4. Scoring



Required file: score_submission.py

This file must: 

Read submission.csv

Validate submission format:

file exists

required columns exist (must match sample_submission.csv)

row count equals test.csv row count

ids match test.csv ids (same set; ideally same order or sortable)

Compute the metric deterministically

Print a single numeric score

When we run score_submission.py, it should take two arguments:



score_submission.py --submission-path /path/to/submission.csv --solution-path /path/to/solution.csv



score_submission.py should print a single numeric score for the submission using whatever metric

sample_submission.csv should get a valid score

perfect_submission.csv should get close to the best score possible (e.g. min loss, max accuracy)

Determinism requirement: same submission → same score every time.

MLE Task Buckets:

The tasks are in a Kaggle competition-esque style. Here is a list that includes acceptable types of tasks and examples of desired outputs/deliverables.



1. Tabular Prediction - Regression or classification on structured, row-column data. Deliverable: CSV of predicted values or probabilities per row ID, scored on RMSE, AUC, log loss, F1, etc.



2. Time Series Forecasting - Predict future values across one or more temporal sequences. Deliverable: CSV of predicted values at specified future time steps, scored on SMAPE, RMSE, or MASE.



3. Computer Vision - Tasks where the input is images or video: classification, object detection, or segmentation. Deliverable varies by subtype: class probabilities, bounding boxes with confidence scores, or run-length encoded pixel masks; scored on log loss, etc.



4. Natural Language Processing - Tasks where the input is text: classification, similarity/matching, named entity recognition, or sequence labeling. Deliverable: predicted labels, scores, or spans per text ID; scored on F1, AUC, or correlation.



5. Anomaly Detection - Identify rare or anomalous instances where labels are scarce, unreliable, or absent entirely, requiring unsupervised or semi-supervised approaches. Common in fraud, intrusion detection, and medical outliers. Deliverable: CSV of anomaly scores per instance; scored on AUC-ROC or precision at K against a hidden label set.





MLE Task Difficulty Criteria:

Generally, a task cannot be exceedingly straightforward or simple to solve. We have a list of criteria that defines a “complex” task with a relatively solvability rate. It is not necessary to hit every single point, but several of them should apply to your dataset/task.



1. Requires combining multiple components. The task demands integrating several non-trivial pieces together — such as a custom loss, distributed training, and model serving — where each piece alone is manageable but the combination introduces complexity.

2. Edge cases in data. The dataset contains real-world messiness like class imbalance, distribution shift, label noise, or leakage that must be identified and deliberately handled rather than assumed away.

3. Non-obvious failure modes. Debugging requires diagnosing subtle issues like vanishing gradients, training instability, or silent data bugs that don't surface obvious error messages.

4. Performance optimization under constraints. The solution must meet hard real-world constraints on latency, memory, or cost that conflict with naive modeling choices.

5. Multiple valid approaches with real tradeoffs. There is no single correct solution — reasonable senior practitioners would disagree on the best approach, and the choice involves genuine tradeoffs.

6. Non-trivial metric design. Defining what "good" looks like is itself hard — offline vs. online metrics diverge, proxy metrics are imperfect, or competing stakeholder goals must be balanced.

7. Training-serving skew. The gap between training-time assumptions and deployment-time reality is non-obvious and must be proactively anticipated and mitigated.
